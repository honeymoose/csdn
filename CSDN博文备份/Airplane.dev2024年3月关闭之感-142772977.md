<p>airplane 这个云服务产品可能很多人都没有用过。</p> <br><p>我们使用的原因是，先前公司非常喜欢使用 airplane 来给运行给已有的产品打运行时补丁。</p> <br><p>前公司的策略就是当发现一个问题可能涉及到数据库的不一致性，那么解决方案就是定时运行一套 SQL 来让数据库保持一致。</p> <br><p>前公司是这个产品的重度使用用户，虽然打补丁这个事情看起来是真的难看，问题根本就没有解决，只是不停的往系统上贴胶布。</p> <br><p>在前公司任职的时候，发现 airplane 这个产品还可以调用 API，因为我们有些任务需要定时运行，与其在后端的 Java 程序中写定时任务还不如通过 API 定时调用来触发任务。</p> <br><p>正因为有这个需求，就发现 airplane 这个云产品还挺好用的，可以直接在其平台上定义一个 API 调用请求，当某个时间点到了后就直接调用这个 API。</p> <br><p>当然，这个 API 调用还可以用 Linux 的 定时任务来执行。</p> <br><p>因为 airplane.dev 有界面，所以感觉还挺好管理和使用的。</p> <br><h3>关闭</h3> <br><p>在2024年10月份的时候，针对我们已部署的产品，突然想去调整下已有的时间，然后找 airplane 网站找了半天都找到，保存的域名也打不开。</p> <br><p>一直以为可能是域名输错了，后来才发现 airplane.dev 的服务于今年 3 月份已经关闭了。</p> <br><p>网上有一个其前员工：Benjamin Yolken 写的博文，在这篇博文中比较详细的描述了 airplane.dev 的关闭过程。</p> <br><p></p> <br><p></p> <br><p class="img-center"><a src="https://img-blog.csdnimg.cn/img_convert/288261fb19007c3f8890f0ac18896cec.png" width="316" /></a></p> <br><p></p> <br><p>其原文位于：<a href="https://yolken.net/blog/end-of-airplanedev" rel="nofollow" title="The end of Airplane.dev | Benjamin Yolken">The end of Airplane.dev | Benjamin Yolken</a></p> <br><p>有兴趣的朋友可以去读下原文，在原文中作者作为一个被迫离职的前员工也表达了对公司决定的不解。</p> <br><p>在当前的大环境下，airplane.dev 的关闭对中国公司的关闭和员工离职，尤其是 IT 行业，有着很多的共性，很有参考价值。</p> <br><h3>关闭原因和前员工看法</h3> <br><p>整个公司的关闭和处理方式可以说是美式标准化的模范生。</p> <br><p>在某一天，公司 CEO 突然告诉大家放下手中工作，有重要的事情宣布，这通常就是公司裁员和关闭的前兆了。</p> <br><p>根据 Benjamin Yolken 的博文，公司在过去的一段时间还是取得了非常大的发展，公司的账上的资金也足够维持一段时间。</p> <br><p>公司关闭的原因是公司的 CEO 将公司出售给了 Airtable 公司，并且现有的员工都会被安排到 Airtable 进行面试。</p> <br><p>已有的 airplane 产品线会被全部抛弃，包括使用 airplane 的所有用户。</p> <br><p>虽然公司还为员工承诺过原始股，当公司关闭时，这些承诺的原始股就没有任何存在的价值和意义。</p> <br><p>Benjamin Yolken 认为 airplane 在关闭所有产品之前可以选择把 airplane 在 Airtable 收购的时候保持单独的产品线运行，或者将 airplane 出售，或者是把 airplane 的所有产品进行开源。</p> <br><p>很遗憾的是 airplane CEO 没有选择上面的任何一种方案而是直接让这个产品消失。</p> <br><p>可能资本家和软件工程师对某一个产品的看法有很大的不同吧。</p> <br><p>通过这篇文章能够比较清晰的了解在资本主义国家对某一个产品的定义或者在公司进行特定决策时候发生的事情。</p> <br><p></p> <br><p></p> <br><p class="img-center"><a
