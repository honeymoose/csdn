<p>在运行测试的时候提示下面的错误：</p> <br><pre><code>org.junit.platform.commons.JUnitException: @BeforeAll method 'protected void com.ossez.edtestbank.tests.QuestionTest.setUp() throws java.lang.Exception' must be static unless the test class is annotated with @TestInstance(Lifecycle.PER_CLASS).<br></code></pre> <br><h2>测试实例生命周期</h2> <br><p>为了允许隔离执行单个的测试方法，并避免由于可变测试实例状态而产生的意外副作用，JUnit在执行每个测试方法之前创建每个测试类的新实例（请参阅下面的讲解，何为测试方法）。这个”per-method”测试实例生命周期是 <strong>JUnit Jupiter</strong> 中的默认行为，类似于JUnit以前的所有版本。</p> <br><p>如果您希望JUnit Jupiter在同一个测试实例上执行所有测试方法，只需使用 <strong><code>@TestInstance(Lifecycle.PER_CLASS)</code></strong> 对您的测试类进行注解即可。当使用这种模式时，每个测试类将创建一个新的测试实例。因此，如果您的测试方法依赖于存储在实例变量中的状态，则可能需要在 <code>@BeforeEach</code> 或 <code>@AfterEach</code> 方法中重置该状态。</p> <br><p><strong>“per-class”模式比默认的”per-method”模式有一些额外的好处。具体来说，使用”per-class”模式，可以在非静态方法和接口默认方法上声明 <code>@BeforeAll</code> 和 <code>@AfterAll（否则@BeforeAll与@AfterAll必须是注解在static的方法上才能生效）</code> 。因此，”per-class”模式也可以在 <code>@Nested</code> 测试类中使用 <code>@BeforeAll</code> 和 <code>@AfterAll</code> 方法。</strong></p> <br><p>如果使用Kotlin编程语言编写测试，则可能会发现，通过切换到”per-class”测试实例生命周期模式，可以更轻松地实现 <code>@BeforeAll</code> 和 <code>@AfterAll</code> 方法。</p> <br><h3>使用静态方法</h3> <br><p>因为下面的方法：</p> <br><pre><code>	@BeforeAll<br>	protected void setUp() throws Exception {<br>		Factory.beginTransaction();<br>	}<br></code></pre> <br><p>没有使用静态方法，所以提示上面的错误，你可用使用静态方法后运行。</p> <br><p>你的测试方法应该不会提示错误。</p> <br><h3>Lifecycle.PER_METHOD</h3> <br><p>在方法中使用</p> <br><pre><code>@TestInstance(TestInstance.Lifecycle.PER_METHOD)<br></code></pre> <br><p>来定义测试。</p> <br><p></p> <br><p><a href="https://cdn.ossez.com/discourse-uploads/original/1X/8db2462ff5c575b07966b400ae2a61723be6cf66.png" rel="nofollow"><img alt="junit5-testing-01" height="500" src="https://i-blog.csdnimg.cn/blog_migrate/eb9f43df0b829908e49711775bfe70d5.png" width="480" /></a></p> <br><p></p> <br><p></p> <br><p>使用上面的方法就可以让你的测试执行了。</p>
